# 设备协议数据库化迁移开发计划

> 状态：draft
> 
> 日期：2026-03-12
> 
> 适用范围：`backend`、`frontend`、`fjbms-uniapp`
> 
> 协议来源：
> - 写协议：`doc/oriigin/device_comm_protocol_write_v2.md`
> - 只读状态协议：`doc/oriigin/device_comm_protocol_basic.md`

## 1. 背景

当前项目中，设备协议相关定义被拆散维护在三处：

- `frontend` 维护 Web 端参数寄存器定义、参数页分类、部分功能位和工厂动作定义
- `fjbms-uniapp` 维护移动端同类定义
- `backend` 维护设备参数权限树静态定义

这导致以下问题：

- 同一协议含义在三端重复维护，更新时容易遗漏
- 协议地址、字节位、bit 位、枚举、权限树文案无法做到单一事实源
- 后续协议版本升级时，无法只通过后端配置完成扩展
- 权限树和设备详情页分类很容易出现不同步

本计划目标是将“协议元数据”统一迁移到后端数据库维护，并通过后端接口分发给 Web 和移动端，实现功能无损迁移。

## 2. 兼容结论

可以兼容将 `doc/oriigin/device_comm_protocol_basic.md` 一并纳入数据库化方案。

但需要明确分两层：

1. 协议元数据层：适合数据库化
- 寄存器地址
- 字节位
- bit 位定义
- key
- 文案
- 单位、缩放、偏移
- 分类
- 权限码
- 控件类型
- 枚举项

2. 二进制解析层：第一阶段仍保留代码兜底
- BLE/UART 帧解包
- 动态长度状态区解析
- 基于串数/温度数量的顺延区间解析
- 仪表盘上的派生字段组装

原因是 `basic.md` 中有两类内容：

- 固定地址、固定长度、固定 bit 位定义
- 动态顺延区间，例如 `0x141~(0x141+S-1)` 电芯电压、后续温度、字符串段等

这类动态区如果第一版就完全改成纯数据库驱动，风险偏高。更稳妥的方式是：

- 第一阶段：数据库接管协议定义和展示元数据
- 解析逻辑仍由现有 parser 使用数据库定义辅助
- 第二阶段：再逐步把动态区模板和 bitfield mapping 也抽到数据库

这样可以兼容 `basic.md`，且不会影响现有状态读取能力。

## 3. 目标

迁移完成后达到以下状态：

- 协议地址和功能 key 以后只在后端数据库维护
- Web 和移动端通过后端 registry 接口获取协议定义
- 后端权限树由数据库生成，不再维护静态 Go 树
- 电池类型枚举、功能位、工厂动作位都来自数据库
- 机构类型设备参数权限改为结构化存储，不再依赖逗号字符串作为唯一数据源
- 在切换期间保留兼容兜底，确保功能无损

## 4. 迁移范围

### 4.1 本次纳入数据库化的内容

- 写协议寄存器定义
- 系统参数定义
- `FUNCTION_CONFIG` 功能位定义
- `0x57A~0x57B` 工厂动作位定义
- 电池类型枚举
- 参数页分类
- 权限树节点
- 只读状态协议中的固定状态寄存器定义
- 只读状态协议中的固定 bitfield 定义

### 4.2 第一阶段暂不完全数据库化的内容

- BLE/UART 二进制解包实现
- 动态顺延区间的完整解析算法
- 仪表盘派生字段计算逻辑

这些逻辑第一阶段保持代码实现，但读取的“定义元数据”改由数据库提供或镜像。

## 5. 目标架构

### 5.1 后端成为协议注册表中心

后端数据库维护协议注册表：

- 协议 profile
- 分组
- item 定义
- 枚举项
- 权限树
- 机构类型权限绑定

后端提供两类接口：

- 注册表查询接口
- 权限配置接口

前端和移动端只做消费和缓存，不再以本地静态文件作为主数据源。

### 5.2 前端和移动端运行时装配

前端和移动端新增“运行时协议仓库”：

- 从后端拉取 registry
- 组装成当前页面可直接使用的结构
- 输出和现在相同的能力接口

例如：

- `getParamDef(key)`
- `listParamsByGroup(groupCode)`
- `getPermissionCode(key)`
- `listFunctionFlags()`
- `listFactoryActions()`
- `getEnumOptions(key)`

这样可以尽量减少页面层改造范围。

## 6. 数据库设计

推荐使用 6 张表。

### 6.1 `device_protocol_profile`

用途：协议主版本表

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint pk | 主键 |
| `protocol_code` | varchar(64) | 协议编码，例如 `BMS_BATTERY` |
| `protocol_scope` | varchar(32) | `write` / `basic` |
| `protocol_version` | varchar(32) | 例如 `v2` / `v251023` |
| `device_type` | varchar(32) | 例如 `BMS_BATTERY` |
| `status` | varchar(16) | `draft` / `active` / `inactive` |
| `source_doc_path` | varchar(255) | 来源文档路径 |
| `checksum` | varchar(64) | 注册表校验值 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

唯一约束：

- `(protocol_code, protocol_scope, protocol_version, device_type)`

### 6.2 `device_protocol_group`

用途：页面分类和权限树分组

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint pk | 主键 |
| `profile_id` | bigint fk | 对应 profile |
| `scene` | varchar(32) | `param_page` / `permission_tree` |
| `group_code` | varchar(128) | 稳定编码，例如 `single`、`advanced.system` |
| `parent_group_code` | varchar(128) | 父分组编码 |
| `label` | varchar(128) | 展示名称 |
| `sort` | int | 排序 |
| `visible` | boolean | 是否启用 |
| `ext_meta` | jsonb | 预留 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

唯一约束：

- `(profile_id, scene, group_code)`

### 6.3 `device_protocol_item`

用途：统一存参数项、状态项、功能位、工厂动作

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint pk | 主键 |
| `profile_id` | bigint fk | 对应 profile |
| `item_key` | varchar(128) | 稳定 key |
| `item_kind` | varchar(32) | `param` / `status` / `function_flag` / `factory_action` / `virtual_status` |
| `label` | varchar(128) | 展示名称 |
| `description` | text | 说明 |
| `page_group_code` | varchar(128) | 参数页分组 |
| `permission_group_code` | varchar(128) | 权限树分组 |
| `permission_code` | varchar(128) | 授权判断码 |
| `access_mode` | varchar(16) | `R` / `RW` / `ACTION` |
| `value_type` | varchar(32) | `u8/u16/u32/str/statusPath/bool/action` |
| `ui_control` | varchar(32) | `input/select/switch/button/readonly` |
| `register_address_start` | int | 起始寄存器地址 |
| `register_address_end` | int | 结束寄存器地址 |
| `byte_selector` | varchar(4) | `L/H` |
| `bit_offset` | int | bit 偏移 |
| `bit_width` | int | bit 宽度 |
| `scale` | numeric(18,6) | 缩放 |
| `offset` | numeric(18,6) | 偏移 |
| `unit` | varchar(32) | 单位 |
| `encoding` | varchar(32) | 字符串编码 |
| `function_code` | int | 扩展读写功能码 |
| `target_address` | int | 扩展目标地址 |
| `status_path` | varchar(255) | 读状态时的解析路径 |
| `sort` | int | 排序 |
| `enabled` | boolean | 是否启用 |
| `preserve_other_byte` | boolean | 写单字节时是否保留另一字节 |
| `ext_meta` | jsonb | 扩展元数据 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

关键约定：

- 普通参数：`item_kind=param`
- 只读状态寄存器：`item_kind=status`
- `0x003E` 功能位：`item_kind=function_flag`
- `0x57A~0x57B` 动作位：`item_kind=factory_action`
- 仪表盘派生字段可后续扩展为 `virtual_status`

### 6.4 `device_protocol_item_enum`

用途：枚举值表，例如电池类型

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint pk | 主键 |
| `item_id` | bigint fk | 对应 item |
| `enum_value` | varchar(64) | 枚举值 |
| `label` | varchar(128) | 枚举名称 |
| `sort` | int | 排序 |
| `enabled` | boolean | 是否启用 |
| `ext_meta` | jsonb | 预留 |

### 6.5 `device_protocol_permission_node`

用途：显式维护权限树节点

原因：

- 权限树不是 item 的简单平铺
- 有分组节点
- 有些叶子节点会合并多个 item
- 叶子节点 value 需要保持稳定

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint pk | 主键 |
| `profile_id` | bigint fk | 对应 profile |
| `node_code` | varchar(128) | 节点编码，例如 `group:single`、`41f`、`factory:enterTestMode` |
| `parent_node_code` | varchar(128) | 父节点编码 |
| `label` | varchar(128) | 展示名称 |
| `node_type` | varchar(16) | `group` / `leaf` |
| `register_address` | varchar(64) | 展示寄存器地址 |
| `permission_codes` | jsonb | 对应一个或多个 permission code |
| `item_keys` | jsonb | 对应一个或多个 item key |
| `sort` | int | 排序 |
| `visible` | boolean | 是否启用 |
| `ext_meta` | jsonb | 预留 |

### 6.6 `org_type_device_param_permission`

用途：机构类型设备参数权限明细表

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint pk | 主键 |
| `tenant_id` | varchar(64) | 租户ID |
| `org_type` | varchar(32) | 机构类型 |
| `permission_code` | varchar(128) | 权限码 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

唯一约束：

- `(tenant_id, org_type, permission_code)`

## 7. 权限码兼容策略

为了功能无损，第一版必须保持当前权限码语义不变。

### 7.1 普通参数

继续使用当前按寄存器地址的权限码：

- 例如 `403`
- `40e`
- `45c`

这样同地址不同字节仍共享权限，兼容现有判断逻辑。

### 7.2 功能位

继续使用：

- `function:insulationHeatingEnabled`
- `function:chargeAllowed`

### 7.3 工厂动作

继续使用：

- `factory:enterTestMode`
- `factory:balanceAllOn`

### 7.4 兼容收益

- 旧权限数据可直接迁移
- Web/移动端权限判断逻辑无需改权限语义
- 过渡期可以新旧并存

## 8. 状态协议纳入方式

`device_comm_protocol_basic.md` 纳入后，分三类处理。

### 8.1 固定只读状态寄存器

这类可直接入库：

- `0x100~0x138` 的固定状态寄存器
- `0x12D~0x137` 的保护/指示/告警状态位域
- `0x122~0x128` 的固定统计值

数据库中用：

- `item_kind=status`
- `access_mode=R`
- `register_address_start`
- `byte_selector`
- `bit_offset`
- `status_path`

### 8.2 动态顺延区间

例如：

- `0x141~(0x141+S-1)` 电芯电压
- 后续 N 路温度
- 硬件型号、组编号、板编码、蓝牙 MAC

第一阶段仍由代码负责具体展开解析，但数据库需存模板元数据：

- 起始地址
- 顺延依据，例如 `seriesCount`、`cellTempCount`
- 展示 label 模板
- 结果 path 模板

建议存入 `device_protocol_item.ext_meta`，例如：

```json
{
  "sequence_mode": "series",
  "count_source": "meta.seriesCount",
  "label_template": "电芯{index}电压",
  "status_path_template": "cell.voltagesMv[{index}]"
}
```

### 8.3 派生字段

例如：

- 平均电压
- 最高/最低电压
- 电压差
- 是否均衡中
- 是否有保护

第一阶段仍保留代码计算，不从数据库完全驱动。

原因：

- 这类值往往是多个寄存器解析后的聚合结果
- 强行数据库化会显著增加第一阶段风险

后续如需要，可增加 `item_kind=virtual_status`。

## 9. 后端接口设计

### 9.1 新增协议注册表接口

#### `GET /api/v1/device_protocols/current`

用途：获取当前有效协议版本

请求参数：

- `device_type`
- `protocol_scope`

返回：

- `protocol_code`
- `protocol_version`
- `checksum`
- `updated_at`

#### `GET /api/v1/device_protocols/current/registry`

用途：获取完整注册表

请求参数：

- `device_type=BMS_BATTERY`
- `protocol_scope=write|basic|all`
- `scene=param_page|permission_tree|all`

返回：

- `profile`
- `groups`
- `items`
- `enums`
- `permission_nodes`

### 9.2 兼容保留的权限接口

#### `GET /api/v1/org_type_permissions/device_param_options`

保持现有接口不变，但内部改为从 `device_protocol_permission_node` 生成。

#### `GET /api/v1/org_type_permissions/device_param_permissions/me`

保持现有接口不变，但内部优先从 `org_type_device_param_permission` 明细表查询。

#### `PUT /api/v1/org_type_permissions/{org_type}`

过渡期支持两种入参：

- 旧字段：`device_param_permissions: string`
- 新字段：`device_param_permission_codes: string[]`

服务端处理原则：

- 如果传新字段，优先使用新字段
- 保存时双写：
  - 老字段 `org_type_permissions.device_param_permissions`
  - 新表 `org_type_device_param_permission`

## 14. 初始化 SQL 的生成策略

不建议手工维护大体量 INSERT。

建议做法：

- 写一个生成脚本
- 从当前三端静态定义中读取结构化数据
- 生成标准 SQL seed

生成源建议：

- `frontend/src/common/lib/bms-protocol/param-registry.ts`
- `fjbms-uniapp/common/lib/bms-protocol/param-registry.ts`
- `backend/internal/service/device_param_permission_tree.go`
- `doc/oriigin/device_comm_protocol_write_v2.md`
- `doc/oriigin/device_comm_protocol_basic.md`

最终提交到仓库的是生成后的 SQL，不在运行时依赖脚本。

## 16. 风险分析

### 风险 1：移动端离线不可用

问题：

- 如果移动端完全依赖在线 registry，弱网或离线场景会影响蓝牙直连调参

应对：

- 必须实现“后端接口 + 本地缓存 + 静态快照”三级 fallback

### 风险 2：权限码兼容失败

问题：

- 如果 permission code 改语义，已有租户权限会失效

应对：

- 第一版保持现有 permission code 规则不变

### 风险 3：状态协议动态区外置过度

问题：

- `basic.md` 动态顺延寄存器和派生状态如果一次性完全外置，容易引入状态展示回归

应对：

- 第一阶段只外置元数据
- 解析算法继续保留在代码

### 风险 4：初始化 SQL 人工录入错误

问题：

- 协议项数量多，手录 SQL 容易出错

应对：

- 采用生成式 seed
- 提供校验脚本比对静态定义和数据库结果

### 风险 5：页面改造范围大

问题：

- 参数页当前有不少按业务写死的数组和动作

应对：

- 先做 runtime registry adapter
- 保持页面调用方式稳定

## 17. 验收标准

满足以下条件才视为迁移完成：

- 写协议和只读状态协议元数据可从数据库完整查询
- Web 和移动端参数页分类来自数据库
- 后台权限树来自数据库
- `BATTERY_TYPE` 枚举、`FUNCTION_CONFIG`、`0x57A~0x57B` 工厂动作均来自数据库
- 机构类型设备参数权限改为结构化明细表存储
- 旧权限数据迁移完成且权限结果不变
- 前后端保留兼容 fallback，不影响现网功能

## 18. 建议的开发顺序

建议严格按下面顺序开发：

1. 建表和 seed
2. 后端 registry 查询接口
3. 后端权限树数据库驱动
4. 权限双写和历史迁移
5. Web runtime registry 接入
6. UniApp runtime registry 接入
7. 清理旧静态定义

## 19. 本计划的直接开发边界

本计划确认后，后续开发以此为准，优先遵循以下原则：

- 不一次性移除所有静态定义
- 先数据库化“定义”，再数据库化“消费”
- 先保证兼容，再做去静态化收尾
- `basic.md` 的动态解析第一阶段不做纯数据库执行，只做元数据接管

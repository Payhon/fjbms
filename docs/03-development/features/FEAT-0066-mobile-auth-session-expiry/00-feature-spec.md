# FEAT-0066 移动端认证失效弹窗去重

- status: review
- owner: payhon
- last_updated: 2026-07-09
- related_feature: FEAT-0066
- version: v0.1.0

## 1. 背景与目标
移动端在本地残留过期 `access_token` 时，启动阶段会并发触发用户信息、设备列表、绑定设备缓存等受保护接口。旧版全局响应拦截器对每个 `401/402/403` 都独立弹出“登录已过期”并跳转登录页，导致用户看到连续 3 次登录框。

目标是将认证失效收敛为一次统一处理：同一轮失效只提示一次、只跳转一次登录页，并避免首个 401 后继续发起必然失败的启动请求。

## 2. 范围
- In Scope：UniApp 全局 API 响应拦截、启动绑定设备缓存刷新、首页用户信息与设备列表加载链路。
- In Scope：公开接口白名单，避免登录、注册、验证码、公开内容、运行配置、升级检查接口误触发登录过期弹窗。
- Out of Scope：后端鉴权语义、refresh token 机制、登录页 UI 改版。

## 3. 用户价值
- 未登录或登录过期时只出现一次清晰提示。
- 登录页不再被多次压栈，减少误操作和重复返回。
- 公开页面与升级检查不再因为认证拦截误提示登录过期。

## 4. 验收标准
- 清空 token 冷启动不弹登录过期提示。
- 写入无效 token 冷启动最多弹一次登录过期提示，确认后只进入一次登录页。
- 首个认证失效响应后首页不再继续请求设备列表。
- 正常登录后首页用户信息、设备列表、绑定设备快照仍可正常加载。

## 5. 风险与约束
- 当前后端 `/api/v1/user/refresh` 仍依赖有效 JWT claims，不能作为过期 token 自动续期入口。
- 公开接口白名单必须保持最小范围，避免登录后接口被错误静默。

## 6. 回滚方案
回滚 UniApp 中 `session-expired` 处理模块、`API/index.js` 拦截器改造、首页和绑定设备缓存的认证失效早退逻辑即可恢复旧行为。

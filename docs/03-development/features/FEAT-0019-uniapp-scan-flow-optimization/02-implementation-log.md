# FEAT-0019 UniApp 扫码流程优化 - 实现日志

## 2026-03-21
- 完成功能规格与技术设计，明确实现边界：
  - 设备类型前缀统一收敛到静态配置文件；
  - UUID 扫码路径保留；
  - 仪表首版复用现有设备详情页，采用临时 BLE 会话模式；
  - BMS 绑定成功后直接进入详情页。
- 完成设备前缀静态配置抽离：
  - 新增 `fjbms-uniapp/common/device-provision/device-prefix.js` 作为 TS/JS 共用真源；
  - `scan-code.ts`、`useAddDeviceActionSheet.ts`、`custom-tab-bar/index.js` 全部改为依赖该模块，不再在业务逻辑内直接判断 `AA/AC`。
- 进一步收敛设备前缀模块实现：
  - 删除 `device-prefix.ts` 同名桥接文件，避免双文件维护；
  - 在 `device-prefix.js` 中补充 JSDoc 类型标注，保证 TS 页面继续获得精确类型提示；
  - `scan-code.ts` 改为通过 `DEVICE_TYPE_BMS | DEVICE_TYPE_METER` 推导设备类型联合类型。
- 完成扫码入口统一分流：
  - 支持二维码解析结果返回 `deviceType`；
  - BMS MAC 继续进入 BLE 搜索/绑定链路；
  - 仪表 MAC 直接进入设备详情页临时 BLE 会话模式；
  - 不支持的 MAC 前缀统一提示“暂不支持该设备二维码”。
- 完成设备绑定成功后的跳转优化：
  - `provision-wizard.vue` 在绑定成功后读取 `bind` 返回的 `device_id` 并 `redirectTo` 设备详情页；
  - `uuid-bind.vue` 同步改为绑定成功后直接进入设备详情页。
- 完成仪表临时 BLE 会话模式：
  - `useBatteryDetail.ts` 新增 `sessionMode = cloud | instrument`；
  - 仪表模式下使用最小本地 `battery` 视图模型建立 BLE 连接，不发起云端详情请求，不上报连接状态/快照，不接 Relay；
  - `detail.vue` 支持解析 `session_mode=instrument&ble_mac=...` 路由参数，并展示仪表临时连接提示卡片。
- 完成仪表详情二次扫码绑定 BMS：
  - `detail.vue` 新增“继续扫码绑定 BMS”按钮；
  - 仅接受 `bms` 类型二维码；
  - 调用现有 `configureMeterMac({ meterAddress: 0xFC, mac })` 写入目标 BMS MAC，并在成功后刷新本地轮询。
- 完成参数页 OTA 显隐与设备类型判断收敛：
  - `params-tab.vue` 新增 `allowOta` 控制，仪表临时会话隐藏 OTA 操作；
  - OTA 目标地址判定改为调用 `isMeterMac()`，移除原先通过 MAC 首字节硬编码判断仪表的逻辑。
- 修复微信小程序端仪表临时连接卡片排版：
  - `detail.vue` 将提示卡片从“文案区 + 按钮横向排列”改为“文案在上、按钮独占一行”的纵向布局；
  - 为按钮增加显式 `customStyle` 宽高与圆角，避免 `u-button` 在小程序端挤压文案区导致中文逐字换行。
- 修复微信小程序端二次扫码绑定 BMS 的运行时与刷新时序：
  - `scan-code.ts` 改为通过 `require('./device-prefix.js')` 读取前缀模块，避免小程序构建器对 CommonJS 命名导入解析不稳定导致的 `parseAddDeviceScanCode` 运行时异常；
  - `detail.vue` 在仪表目标写入成功后，不再只做 `pause/resume` 轮询，而是主动断开当前仪表蓝牙并在短延迟后重建临时会话；
  - 新增 `meter-session` 调试日志，覆盖扫码解析、目标写入、断开重连三段关键链路，便于继续定位协议时序问题。
- 根据真机测试结果调整二次扫码绑定 BMS 的超时判定：
  - 现场日志显示 `configureMeterMac` 在微信小程序端可能超时，但硬件仪表已实际切换到新的 BMS 目标并显示数据；
  - `detail.vue` 现已在发送前先暂停轮询并短延迟避开并发请求；
  - 若 `configureMeterMac` 返回 `BLE request timeout`，前端不再直接判定为失败，而是视为“指令可能已生效”，继续执行断开并重连仪表会话，等待后续状态读取恢复。
- 优化仪表临时连接浮层交互：
  - 临时连接提示卡片改为覆盖在仪表盘顶部的浮动半透明层，不再占据正常页面布局高度；
  - 浮层支持收起，收起后在顶部蓝牙连接区域右侧显示仪表入口图标，可随时重新展开浮层；
  - 浮层仅在仪表临时会话且位于仪表盘标签页时显示。
- 优化仪表临时会话首包显示速度：
  - `useBatteryDetail.ts` 将仪表临时会话的轮询从固定 `5s` 间隔调整为“未拿到首个 `status` 前使用约 `1.2s` 的短间隔重试，首包成功后恢复常规轮询”；
  - 重新连接仪表会话时，将重连延迟从 `900ms` 缩短到 `450ms`，减少扫码绑定 BMS 后重新看到数据的等待时间。
- 修正微信小程序端浮层定位：
  - `detail.vue` 将仪表临时连接浮层的锚点改为基于 `navHeight` 计算，确保浮层稳定显示在“蓝牙连接 / 断开”行下方；
  - 避免浮层按页面绝对顶部定位，遮挡返回栏与页面标题。
- 修正浮层层级与顶部仪表入口对齐：
  - 仪表临时连接浮层改为 `fixed` 浮动层并提升层级，确保显示在仪表盘组件上方，不被内容卡片遮挡；
  - 小程序顶部的仪表入口按钮尺寸调整为与“蓝牙连接 / 断开”胶囊同高，并在连接行内垂直居中对齐。
- 兼容微信小程序 `canvas` 原生层级：
  - 现场验证发现仪表盘主体使用 `canvas` 绘制，普通 `view` 浮层即便提升 `z-index` 也可能被原生层压住；
  - `detail.vue` 现已在 `MP-WEIXIN` 下将仪表临时连接浮层切换为 `cover-view` 实现，保证浮层稳定覆盖在仪表盘 `canvas` 上方；
  - 其它平台继续保留常规 `view` 浮层实现。
- 调整仪表临时连接浮层的显示时机：
  - 仪表临时连接页面在蓝牙仍处于 `connecting` 阶段时，不显示浮层和顶部仪表入口；
  - 仅当蓝牙已成功连接到仪表后，才展示浮层或其收起后的入口按钮，减少连接过程中的视觉遮挡。
- 优化仪表页 BLE 连接排队时延：
  - 现场日志显示，当账号设备列表中已有被仪表代理的 `AC` BMS 设备时，旧详情页残留的 BLE 自动连接任务可能继续尝试连接 `AC...` 设备；
  - BLE 缓存层原先使用全局串行锁，导致新开的 `AA...` 仪表连接任务必须等待旧任务完成直连重试与 `5s` 扫描，表现为“首次连接很慢且先失败”；
  - 现已在 `ble-client-cache.ts` 引入连接任务失效 epoch，并在 `useBatteryDetail.ts` 的 `disconnectAll()` 中主动使旧连接任务失效，切页后过期的 `AC...` 连接任务会尽快中止，不再阻塞新的仪表连接。

## 2026-03-25
- 新增“已添加设备直达详情”扫码分支：
  - `bound-devices.ts` 扩展 `item_uuid` 字段与 `findByBleMac / findByItemUuid` 查找方法；
  - 新增 `common/device-provision/scan-routing.js` 作为扫码跳转决策模块，统一输出“已绑定详情 / BMS 添加 / 仪表临时会话 / UUID 绑定”四类动作；
  - `useAddDeviceActionSheet.ts` 与 `custom-tab-bar/index.js` 均改为在扫码后先刷新绑定设备列表，若命中已添加设备则直接跳转 `/pages/device-battery/detail?device_id=...`，不再进入添加流程。
- 兼容微信小程序 `custom-tab-bar` 的模块加载限制：
  - 现场编译日志显示 `custom-tab-bar/index.js` 直接 `require()` 外部 `common/*.js` 模块存在运行时未定义风险；
  - 现改为在 `main.ts` 启动时把设备前缀配置写入 storage，并由 `bound-devices.ts` 在刷新后同步维护绑定设备快照；
  - `custom-tab-bar/index.js` 仅读取 storage 中的前缀配置和绑定设备快照做扫码跳转决策，避免跨目录 `require()` 导致的小程序运行时报错。

# FEAT-0027 后台删除终端用户后账号身份残留修复 - 功能规格

- status: review
- owner: payhon
- last_updated: 2026-03-26
- related_feature: FEAT-0027
- version: v0.1.0

## 1. 背景与目标
当前后台管理删除终端用户时，仅删除 `users` 主记录，未同步删除 `user_identities`、`device_user_bindings` 等 APP 账号关联数据。结果是后台列表中已找不到原账号，但手机号、邮箱、微信 `openid` 仍被残留身份记录占用，导致用户重新注册后继续提示“账号已被绑定”。

本次需要修复后台删除用户的级联清理逻辑，并补一条历史数据清理 SQL，消除线上已有孤儿记录。

## 2. 范围（In Scope / Out of Scope）
In Scope:
- 后台 `DELETE /api/v1/user/{id}` 删除用户时，事务清理 APP 账号关联表。
- `user_identities` 查询新增“孤儿身份自修复”，遇到引用不存在用户的身份记录时自动视为不存在。
- 新增 SQL 补丁，清理历史孤儿 `user_identities` / 绑定记录。
- 同步功能文档与项目看板。

Out of Scope:
- 不改造移动端绑定 UI。
- 不扩展新的账号合并、迁移或人工找回流程。
- 不处理与本问题无关的普通组织账号业务逻辑调整。

## 3. 用户价值
- 后台删除终端用户后，手机号/邮箱/微信身份可以重新注册和重新绑定。
- 平台数据口径恢复一致，不再出现“后台查不到用户但绑定仍被占用”的异常。
- 线上历史脏数据可通过标准 SQL 补丁一次性修复。

## 4. 验收标准
- 后台删除终端用户后，相关手机号重新注册成功。
- 重新注册后的账号可正常绑定手机号、邮箱、微信，不再提示“已被绑定”。
- 历史执行 SQL 补丁后，已有异常账号恢复正常。
- 文档与 `docs/04-project-tracking/board.md` 已同步。

## 5. 风险与约束
- 删除用户是破坏性操作，级联清理必须在事务内执行，避免中间状态残留。
- 历史数据库可能已存在孤儿 `device_user_bindings`、`app_device_added_records`、`message_push_*` 等记录，需要一并清理。
- 需保持对现有组织账号删除逻辑兼容，避免误删非用户主键关联之外的数据。

## 6. 回滚方案
- 回滚后端后台删用户级联清理与 `GetUserIdentity` 自修复逻辑。
- 若已执行 SQL 46，数据清理不可逆；回滚仅限代码层恢复旧逻辑。

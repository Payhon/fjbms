## 1. 背景与目标
当前系统的 `users` 仅有 `name`、`phone_number`、`email` 等字段，其中：
- APP / 小程序账号体系里，登录标识依赖手机号或邮箱；
- 移动端设置页又把 `name` 同时承担“昵称”和“用户名/账号名”语义；
- 后台用户管理和终端用户列表也没有独立展示账号名字段。

这导致几个问题同时存在：
- 纯微信小程序直注册账号没有清晰的“账号”概念，用户无法确认自己的登录账号；
- 已存在用户缺少统一的 `username` 字段，后续账号展示、换绑联动和后台查询都缺乏稳定口径；
- 移动端“我的”页手机号显示依赖错误字段，出现“左上角显示未设置手机号，但设置页实际已绑定”的异常；
- 后台表格只能看到姓名/昵称，无法直接看到账号名。

本次需要为 `users` 新增独立 `username` 字段，并贯通历史补齐、注册默认值、手机号换绑联动、移动端设置页展示和后台表格展示。

## 2. 范围（In Scope / Out of Scope）
In Scope:
- `users` 表新增独立 `username` 字段，并建立租户内大小写不敏感唯一约束。
- APP / 小程序注册流程按账号类型自动补齐默认 `username`。
- 历史用户数据通过 SQL 补丁补齐 `username`。
- 移动端“我的 / 设置”新增账号展示、设定账号入口、手机号换绑能力，并修复手机号显示 bug。
- APP / 小程序密码登录在输入纯数字账号时，优先按 `username` 查找用户，未命中时再按 `phone_number` 回退。
- 后台 `bms/system/user`、`app_manage/users`、`bms/end_user` 三处用户相关表格新增“用户名”展示。
- 文档、SQL、接口返回、Swagger 和前端类型同步更新。

Out of Scope:
- 不新增任意格式用户名登录能力；仅对“纯数字账号”补充 `username -> phone_number` 的解析顺序调整。
- 不开放后台直接修改 `username`，后台仅只读展示。
- 不改造 Web 后台的旧模板页 `manage/user*`。
- 不引入账号合并、账号迁移、批量人工修正等运营工具。

## 3. 用户价值
- 终端用户拥有清晰稳定的账号名字段，可区分“昵称”和“登录账号”。
- 纯微信小程序账号可以在设置页补设账号，后续找回或账号识别更明确。
- 手机号换绑后，账号名在需要时自动跟随更新，减少账号信息错乱。
- 后台运营和客服在用户表格中可直接看到用户名，定位账号更高效。

## 4. 验收标准
- `users` 表存在 `username` 字段，并对非空值生效租户内唯一约束。
- 手机号注册用户默认 `username = phone_number`；邮箱注册用户默认 `username = email`；纯微信直注册保持空。
- 历史执行 SQL 补丁后，已有用户的 `username` 按规则补齐，冲突数据可自动规避唯一索引冲突。
- 移动端“我的”页对已绑定手机号用户不再错误显示“未设置手机号”。
- 移动端设置页在“修改密码”上方新增“账号”行；纯微信直注册且无账号时可进入“设定账号”页成功保存。
- 已绑定手机号用户点击手机号后可完成换绑；若旧 `username` 等于旧手机号，则换绑成功后自动更新为新手机号。
- 密码登录输入纯数字账号时，如存在同名 `username`，应优先命中该用户；仅在 `username` 未命中时才按手机号查询。
- 后台 `bms/system/user`、`app_manage/users`、`bms/end_user` 三张表均显示“用户名”列。
- 文档与 `docs/04-project-tracking/board.md` 已同步。

## 5. 风险与约束
- `username` 与现有 `name` 语义必须彻底分离，避免昵称功能回归时误写账号名。
- 历史数据补齐需要同时考虑占位邮箱、空手机号以及租户内冲突，补丁必须可重复执行。
- 手机号换绑涉及 `users` 与 `user_identities` 两张表，必须在事务内同步更新。
- 后台表格新增字段后会增加横向宽度，需要同步调整列宽和滚动宽度，避免遮挡操作列。

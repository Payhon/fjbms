# 文档使用原则（fjbms）

## 1. 目标
通过文档治理确保开发过程可追溯、可协作、可交付。文档是开发入口，不是附属品。

## 2. 强制原则
1. **先文档，后开发**：任何功能开始编码前，必须先创建 `00-feature-spec.md`。
2. **文档与代码同步**：每次实现变更，必须同步更新对应文档与 `docs/04-project-tracking/board.md`。
3. **完工四要素**：功能标记完成前，必须同时满足代码、测试、文档、看板状态四项一致。
4. **单一事实源**：功能状态以 `board.md` 为唯一状态源，功能目录文档为细节源。
5. **默认中文**：文档默认中文，必要术语可中英并列。

## 3. 文档生命周期
`draft -> approved -> in_progress -> review -> done -> archived`

## 4. DoD（完成定义）
功能进入 Done 前必须满足：
- 代码完成并合并
- 测试报告已更新
- 发布说明已更新
- 看板状态同步为 Done
- 无未记录阻塞
